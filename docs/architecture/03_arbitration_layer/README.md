# 03 仲裁层

## 概述

仲裁层是引擎的**可行性裁判**，负责判断一个原子动作在当前状态下是否**物理和逻辑上可能发生**。
它通过**单次 LLM 调用**完成所有维度的评估，输出综合检查报告。
不通过时，由 LLM 直接生成叙事内拒绝文本——玩家永远不会看到系统语言。

### 核心原则：玩家自由

**仲裁层只拦截物理/逻辑上不可能的行为，绝不拦截社交失当或"不明智"的行为。**

这是一个自由扮演的 CRPG——玩家可以扮演任何性格：鲁莽、粗鲁、荒诞、恶棍、喜剧。
社交后果由世界模拟承担（NPC 愤怒、卫兵介入、关系恶化），而不是由仲裁层阻断行为。

---

## 四维评估

仲裁层在一次 LLM 调用中评估以下四个维度：

### 第一维：信息完整性
角色当前是否拥有执行此动作所需的信息？

- 角色是否知道目标的位置？
- 角色是否认识目标人物？
- 角色是否了解执行动作所需的前提知识？

> 关键原则：**玩家知道的不等于角色知道的。**

### 第二维：物理/空间可行性
物理/空间层面是否可行？

- 角色当前位置与目标位置之间是否存在可行路径？
- 是否有物理阻断（锁着的门、敌对区域、时间不对）？
- 场景中未明确提及但合理存在的物品应视为存在（酒馆有桌椅、森林有树石），只有明显不可能的才拒绝

### 第三维：逻辑一致性
此行为是否与已确立的世界事实矛盾？

- 是否在与已死亡的角色对话？
- 是否使用已消耗的物品？
- 是否出现其他事实性悖论？

### 第四维：叙事轨道检查
此动作是否会导致叙事严重偏轨？

- 是否会直接跳过多个关键节点？
- 是否触发轨道修正阈值？（详见 [05 叙事轨道](../05_narrative_rail/README.md)）

> **第四维永远不会导致拒绝**——它只标记 `drift_flag`，由异步的叙事轨道 Agent 决定干预方式。

---

## 检查结果处理

### 通过
动作传入事件 Agent 生成对应事件。
若携带"强行推进"标记（来自反思系统坚持机制），事件 Agent 在生成时附加负面后果权重。

### 不通过
LLM 在同一次调用中直接给出叙事内拒绝文本。
**任何情况下，都不向玩家暴露"检查失败"等系统语言。**

---

## 社交行为的处理（不拦截，承担后果）

社交失当的行为（如在国王面前胡言乱语）**不会被仲裁层拒绝**，而是：

1. 仲裁层放行（物理/逻辑上可以说话）
2. 事件生成器忠实执行玩家行为
3. 世界产生合理后果（国王暴怒、卫兵拔剑、关系破裂）

这确保了 CRPG 的核心体验：**选择权属于玩家，后果属于世界。**

---

## 叙事内拒绝

拒绝必须发生在世界内部。LLM 根据失败的维度和上下文，自由选择合适的拒绝方式，例如：

### 叙事内消化
通过角色的认知或内心声音体现障碍，不出现系统提示。

> "你意识到你甚至不知道他在哪里。城里有人提到过他的名字，仅此而已。"

### 部分执行 + 自然中断
能执行的部分正常执行，不能执行的通过世界反应自然阻断。

> "你向城北走去——那是你仅有的线索。走到一半，你停下来。你准备好了吗？"

### 意图重新解读
对语义模糊的意图，不拒绝，而是呈现可能的路径并说明当前条件不足。

> "让他倒下——但你还没有筹码。说服需要他信任你，而他现在只知道你是个陌生人。"

### 代价式执行（仅用于坚持情形）
玩家明确坚持时，不阻止，但执行附带真实代价。

> "你冲进了他的办公室。结果是灾难性的，但这是你选择的。"

---

## LLM 输出格式

```json
{
  "passed": false,
  "checks": [
    { "dimension": "information_completeness", "passed": true, "reason": null },
    { "dimension": "physical_spatial", "passed": false, "reason": "市长办公室的门上着锁" },
    { "dimension": "logical_consistency", "passed": true, "reason": null },
    { "dimension": "narrative_drift", "passed": true, "reason": null }
  ],
  "drift_flag": false,
  "rejection_narrative": "市长办公室的门紧锁着，你没有钥匙无法进入。"
}
```

## Pipeline 实现

### 设计原则

- 四维评估合并为**单次 LLM 调用**，减少延迟和 token 消耗
- 所有上下文（主观记忆、客观状态、Lore、近期事件）在调用前并发查询
- LLM 自由判断各维度是否通过，不通过时直接生成叙事内拒绝文本
- 无硬编码枚举——动作类型、拒绝策略均由 LLM 语义判断

### 步骤序列

```
输入：AtomicAction（单个原子动作）+ PipelineContext

// 并发查询阶段（代码）
ParallelQueryStep:
  Promise.all([
    主观记忆（memory:subjective:{characterId}）    → 用于信息维度
    客观世界状态（world:objective:{characterId}）   → 用于物理/空间维度
    Lore 相关条目（按 action.target 查询）          → 用于逻辑一致性维度
    近期事件历史（最近 10 条 Tier1 标题）            → 用于叙事漂移维度
  ])

// 单次 LLM 评估
FeasibilityCheckStep:
  将动作 + 所有上下文传给 LLM，要求评估四个维度：
    1. 信息完整性
    2. 物理/空间可行性
    3. 逻辑一致性
    4. 叙事漂移（仅标记，不拒绝）

  LLM 返回综合报告：
    ├─ passed: false → 报告中包含 rejection_narrative → 短路返回
    └─ passed: true → 继续
        drift_flag 写入 context，由异步叙事轨道 Agent 处理

// 汇总结果
ArbitrationResultStep:
  组装 ArbitrationResult { passed, action, force_flag, force_level, drift_flag }
  传递给事件 Pipeline

输出：ArbitrationResult
```

### LLM 输入

```json
{
  "action": { "type": "MOVE_TO", "target": "mayor_office", "method": null, "order": 0 },
  "subjective_memory": { ... },
  "objective_world_state": { ... },
  "lore_context": [ ... ],
  "recent_events": ["事件标题1", "事件标题2"]
}
```
